Mario Clone (2019)

Project Info:

I developed this project for my Game Engines Creation module during my first year of university, programming it in C++ using the SDL library. The end result is a clone of both the original arcade Mario Bros game, and the original Super Mario Bros for the NES, combined into one game. I earned a grade of 97% on the assignment this project was developed for.

Read More

Features

  • Engine Features:
    • AABB Collision with blocking capabilities
    • User interface with working text and buttons
    • Sprite animations
    • A level editor that can save, load and edit levels
    • Music and sound effects
    • A system for managing and switching between levels
  • Gameplay Features:
    • Level victory (for Super Mario Bros) and death (for both)
    • Two player mode for Mario Bros
    • Working koopas with kickable shells that cause damage
    • Coins that give score
    • Breakable blocks, pow blocks and blocks that give coins
    • Seperate high score tables for both games with a screen to enter name after level is finished

What went well?

  • Both games were implemented successfully and strongly resembled the source material
  • Making the level editor, main menu and high score tables taught me a lot about developing UI components from scratch
  • My level editor was robust enough to enable me to accurately recreate the levels from the source material
  • Through developing this project I was able to attain a deep understanding of how to use the SDL library
  • In developing the engine for this project I was able to build on my knowledge of OOP to organize my functionality in a much more clean fashion
  • My knowledge of file I/O was boosted significantly in implementing saving and loading for high scores and levels
  • My system for managing and switching between levels ended up working really well
  • Multiple player characters for two player was easy to implement because of good design

How could I improve it?

  • My collision blocking had minor unexpected behaviour in a few edge cases
    • With more time, I would have researched a more robust method of collision blocking
  • The level editor could have been more user-friendly
    • More hotkeys could have boosted workflow efficiency
  • My implementation for UI buttons could have been improved
    • The current system is not very modular and adding new buttons can be a very complicated task
    • Performing AABB checks on every button onscreen to see if it collides with the mouse every frame is computationally expensive
  • I could have added more features to make the recreation more accurate
    • This include enemies such as goombas, and powerups such as stars, mushrooms and fire flowers
  • I could have added more possible characters for text

DirectX 11 Graphics Project (Work in Progress) (2019-)

Project Info:

This is a work in progress application I am developing in DirectX 11 for my Further Games & Graphics Concepts module in my second year of university. The plan is to make a 3D aquatic boat simulation game with working graphics, and then eventually I intend to implement physics next semester. I have been working on this project for the last 2 months and hope to have it finished by the end of the academic year.

Current Features

  • Specular, diffuse and ambient lighting
  • Texture mapping, specular mapping
  • Third person camera movement
  • .obj file loading
  • Hierarchical scene graphs
  • Skybox

Planned Features

  • User interface with an external library
  • Fixed first person camera movement
  • Controllable boat with a camera that follows it
  • Water effects
  • Level data storage with JSON
  • Point, directional and spot lighting

What is going well?

  • Lighting and texturing are working exactly as intended
  • Lighting makes use of per-pixel phong shading successfully
  • Camera movement works well in third person
  • The hierarchical scene graph works and is very easy to use, and it is very easy to add new children to objects
  • The skybox works well and follows the camera
  • My game objects support having varying meshes, textures, specular maps, vertex shaders, and pixel shaders
  • I am developing a strong understanding of working with DirectX 11, and shader programming with HLSL
  • Functionality is very well organized into object oriented data structures to enable modularity

How do I need to improve?

  • The skybox needs a better texture (currently placeholder), and the shape needs to be changed from a cube to a sphere
    • Because the cube I am using is not intended to have inverted normals there is a weird black mark in the upper and lower portions of the shape
  • The camera should use interpolation to make movement less jarring
  • I need to implement the first person camera that follows objects and locks to their position and moves around them
  • The camera rotation gets buggy when looking straight up or straight down, so I need to correct that.

Tilewar (2018)

Project Info:

Tilewar was a project I developed at college during my studies in game design for a specialist project, when I specialized in engine development and programming. It is a turn based strategy game made in UE4 primarily with the C++ API it provides, and features a custom implementation of the A* pathfinding algorithm.

Read More

Features

  • A custom tile grid system
  • A custom implementation of A* that is compatible with my tile system
    • Makes use of UE4's binary heap data structure
    • Uses euclidean distance and tiebreaker heuristics
  • Multiple levels
  • Basic graphics settings
  • AI that reacts to conditions in the game (defensive when hurt, otherwise on the offensive against the player)
  • Multiple player characters that the player can switch between and control if they have usable movement points in the turn
  • Win and lose conditions based on if the player kills all the enemies or the enemies kill all player characters

What went well?

  • In developing this project I learned how to use C++ with Unreal Engine 4 and developed my skills in working with existing complex codebases
  • I successfully taught myself the A* pathfinding algorithm and managed to implement it into UE4 whilst adapting to and working with the constraints and pre-existing functionality of the engine
  • I managed to get the AI working well and in a way that feels challenging and interesting to play against

How could I improve it?

  • There are some minor bugs with turn order that could be fixed if I had more time to work on the project
  • Visuals could do with some work
  • The game would perhaps be more interesting if instead of using a tile based system it instead was more freeform with some terrain that takes longer to traverse than other terrain
  • My calculations that work out damage dealt by attacks that factor in the attack and defense of both the attacker and defender could have been more thought out
